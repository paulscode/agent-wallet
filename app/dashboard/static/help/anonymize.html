Anonymize — troubleshooting

What each awaiting_reconciliation reason means, what's happening to your funds, and what to do about it. This is the dashboard-served mirror of the canonical docs/anonymize.md troubleshooting section.

Per-reason index
  • Lightning routing failed B
  • Network throttled — retrying A
  • Hit a snag A
  • Session went stale A
  • Session schema too old C
  • Can't reach operator A
  • Can't read on-chain fees A
  • Lightning payment stuck B
  • Operator changed fees B
  • Operator security check failed C
  • Couldn't sign claim C
  • Clock too far off C
  • Can't receive over Lightning A
  • Unknown reason C

Class legend

Each reason maps to a recovery class. A reasons are transient and the wallet auto-retries on a backoff schedule. B reasons get a small bounded retry budget (3 attempts) before escalating, because they carry funds-state implications. C reasons need operator action — the wallet refreshes the "last seen" timestamp but doesn't retry.

Operator decision tree

  1. Open the row's Details disclosure on the dashboard. Note the raw reason code and the pre-reconciliation status (Was: line).
  2. Find the per-reason block below.
  3. Click Try again if the underlying condition has cleared.
  4. Click Refund if funds are recoverable but the session can't safely resume. Requires step-up nonce confirmation.
  5. Click Cancel when no funds moved and you want to abandon.
  6. Inspect the session's audit log (anonymize_session_event rows) for the full sequence of reconciliation_attempt_* / reconciliation_escalated events.

Lightning routing failed B mpp_k_floor_exhausted

What it means
The reverse-leg's Lightning payment couldn't find a route to Boltz at any of the configured MPP K values. By default the wallet tries K=4, K=3, K=2; all three failed with FAILURE_REASON_NO_ROUTE.
Funds
Zero sats moved. NO_ROUTE means LND never committed an HTLC. Refunding isn't needed and isn't offered.
Wallet does
Auto-retries up to 3 times on backoff. On exhaustion, escalates to failed.
You can
Try again if you've added outbound LN liquidity or expect the routing landscape has changed. Cancel to abandon (safe — no funds moved).
Escalate when
Repeated mpp_k_floor_exhausted across many sessions usually points to outbound-liquidity structure. Fix by adding LN channels, rebalancing, or adding smaller values to ANONYMIZE_AMOUNT_BINS_SAT.

Network throttled — retrying A circuit_rebuild_throttled

What it means
The Tor circuit-rebuild token bucket blocked another attempt. The wallet rate-limits these to deny entry-guard fingerprinting during operator outages.
Funds
Zero sats moved. The throttle fires before any HTLC could be committed.
Wallet does
Auto-retries every probe tick (default 5 min) until the bucket refills. Typical recovery within ~60 min.
You can
Watch for the "Session resumed" toast — usually no action needed. Cancel to abandon (safe).
Escalate when
Repeated throttling across many sessions. Check Tor entry-guard reachability, Boltz operator availability, or bump ANONYMIZE_CIRCUIT_REBUILD_TOKENS_PER_HOUR.

Hit a snag A bounded_retry_exhausted

What it means
The per-session loop's tick handler raised an exception too many times in a row (bounded-retry counter). The wallet routes the session to reconciliation rather than hammering a wedged dependency.
Funds
Zero sats moved at the threshold. The counter only increments on exceptions before any fund-moving step, so by construction this is pre-payment.
Wallet does
Auto-retries until the underlying exception clears (typically a transient dependency: DB, LND REST, Tor circuit).
You can
Try again to force a fresh attempt. Cancel to abandon (safe).
Escalate when
The same session repeatedly lands here. Inspect the container logs around the session id for the underlying exception. Likely: LND connectivity drift, DB lock contention, or an orchestrator bug.

Session went stale A wall_clock_budget_exceeded

What it means
The wedge detector noticed a non-terminal row whose updated_at is older than ANONYMIZE_RECONCILIATION_RUNTIME_WALL_CLOCK_BUDGET_S (default 4 h). Catches sessions that wedged in an active status without self-reporting a failure.
Funds
Depends on pre-status. Open Details and read the Was: line:
  • sending to your address (exiting) — may be in flight; Refund available.
  • mixing (hopping) — may be in flight; Refund available.
  • received funding or earlier — no funds moved.
Wallet does
Auto-retries the resume path on backoff.
You can
Wait for auto-retry (countdown on the row). Refund if pre-status indicates funds in flight. Cancel if pre-payment.
Escalate when
Sessions repeatedly time out at this stage — likely an orchestrator bug where a hop body returns from a tick without forward progress.

Session schema too old C pipeline_schema_below_min_supported

What it means
The session row was created under an older wallet version whose pipeline_json shape the running code no longer executes. The schema-version gate refuses to drive forward.
Funds
Depends on pre-status — same heuristic as wall_clock_budget_exceeded.
Wallet does
No automated retry. The row stays parked indefinitely.
You can
Mark done if pre-status is pre-payment. Refund if pre-status indicates funds in flight.
Escalate when
A wallet upgrade has bumped ANONYMIZE_MIN_SUPPORTED_PIPELINE_SCHEMA_VERSION past in-flight session schemas. Usually the right action is refund every affected session, then start fresh.

Can't reach operator A external_state_unknown

What it means
The wallet couldn't query Boltz (or the chain backend) for the session's external state. Common causes: Boltz onion endpoint is down, Tor circuit churn, deep chain-backend lag.
Funds
May be in flight. Boltz may already be holding LN funds or have published the on-chain claim. Until the external state can be queried, the wallet doesn't know.
Wallet does
Auto-retries via the polling cadence + circuit-rebuild path.
You can
Wait — most outages recover within minutes.
Escalate when
Class A budget exhausts → wallet escalates to failed. At that point: verify Boltz reachability, check the operator-registry signature, reconcile from Boltz's database manually if needed.

Can't read on-chain fees A economy_feerate_unavailable

What it means
The feerate sanity gate couldn't get a fresh economy feerate from the chain backend after two attempts. The wallet refuses to claim against an outlier fee.
Funds
In flight. The reverse swap's LN side may already be paid. The claim is paused.
Wallet does
Auto-retries the feerate probe.
You can
Wait — usually resolves within one probe tick. Refund if the swap's timeout_block_height is approaching.
Escalate when
Chain backend (electrs / mempool HTTP) is persistently failing. Inspect the chain_poll recurring task's health.

Lightning payment stuck B stuck_htlc_alarm

What it means
The stuck-HTLC alarm fired — your reverse-leg LN payment has been in-flight at LND for longer than the documented threshold without settling or failing.
Funds
In flight on Lightning. LND holds the HTLC; Boltz hasn't received the preimage yet.
Wallet does
Bounded auto-retry by re-polling LN payment state.
You can
Wait — most stuck HTLCs settle or fail within the channel's CLTV expiry window. Refund if you don't want to wait (LN fees already paid are not recoverable).
Escalate when
Repeated stuck-HTLC alarms across sessions point to a routing peer that's accepting HTLCs but not forwarding them — consider adding that peer to ANONYMIZE_AUTO_BLOCKLIST_TOP_N_PEERS.

Operator changed fees B claim_feerate_outlier

What it means
Boltz returned a claim feerate outside the configured tolerance band relative to the wallet's live economy estimate. The wallet refuses to claim at that feerate.
Funds
In flight at the operator. The ANONYMIZE_CLAIM_FEERATE_OUTLIER_GRACE_S grace window allows re-quoting before escalating to refund.
Wallet does
Bounded auto-retry by re-requesting a fresh quote. On grace exhaustion, escalates to refunding.
You can
Wait. Refund to take the on-chain refund path immediately.
Escalate when
Repeated outlier feerates from one operator mark them degraded_operator. Consider removing them from your operator pair set.

Operator security check failed C operator_signature_mismatch

What it means
A response from a Boltz operator failed the signature check against the registry's pinned public key. This is a security-critical event — either the operator rotated keys without coordinating a registry update OR the response was tampered with in transit.
Funds
In flight. Boltz has your LN payment; the wallet is refusing to act on the operator's claim response.
Wallet does
No automated retry. Retrying would mask an active attack.
You can
Refund — recommended. The on-chain refund path uses the swap's locally-held refund_private_key, not the operator's signature, so it's safe even if the operator key is compromised.
Escalate when
Always. Check for a recent operator-key rotation announcement; if none, treat as a potential incident.

Couldn't sign claim C claim_tx_validation_failed

What it means
The cooperative-claim Musig2 ceremony produced a transaction that failed the wallet's local validation gate. The wallet refuses to broadcast it.
Funds
In flight. Boltz has your LN payment; the on-chain claim hasn't broadcast.
Wallet does
No automated retry. Re-running the same Musig2 ceremony with the same inputs would produce the same outcome.
You can
Refund — recommended. The refund path bypasses Musig2 entirely.
Escalate when
Always. Capture the redacted last_error from the Details panel before clicking Refund.

Clock too far off C clock_skew_exceeds_deadline_margin

What it means
The clock-skew gate refuses to drive forward because the host's clock has drifted past the safe deadline margin. Allowing the session to continue would risk broadcasting a claim after the swap's timeout_block_height.
Funds
In flight. Boltz has your LN payment.
Wallet does
No automated retry until the operator confirms the clock is correct.
You can
Verify the host's chrony / NTP status; after the clock catches up, click Try again. Refund if the clock issue can't be resolved before the swap timeout.
Escalate when
The clock has been corrected but the session won't resume — the wallet may need a process restart for the clock-skew probe to re-measure.

Can't receive over Lightning A inbound_insufficient_at_lockup

What it means
This is an on-chain-sourced session. To convert your on-chain funds to Lightning it uses a submarine swap, which requires your node to receive the amount over Lightning from the swap provider. Just before locking the funds on-chain, the wallet re-checked your inbound capacity and found it can no longer cover the amount (inbound that was sufficient when you started the session has since dropped). It aborted before moving any funds.
Funds
Safe — nothing moved. The check fires before the on-chain lockup broadcast, so no coins left your wallet.
Wallet does
Routes the session to reconciliation. Because there's no automatic way to resume an on-chain funding step, the recovery probe escalates the row to Failed rather than retrying.
You can
Cancel right away (no funds moved), then start a new session once you have more inbound capacity — e.g. after a pending channel confirms or a rebalance lands. Alternatively use a Lightning source, which needs no inbound.
Escalate when
You believe you do have ample inbound — check that your channels are active and not fully depleted on the receive side.

Unknown reason C unrecognised

What it means
The reason code on the row isn't in the classifier's known set. The wallet treats unknown reasons as Class C so a new bug-driven reason can't silently retry into a tight loop.
Funds
Status unknown. Open the Details panel for the raw reason string, pre-reconciliation status, and redacted last_error. The combination usually tells you whether funds moved.
Wallet does
Refresh "last seen" on each probe tick but no automated retry.
You can
Try again if the audit log convinces you the reason is transient. Refund if pre-status indicates funds in flight. Stop trying to mark failed once you've decided.
Escalate when
Always — this means either a wallet upgrade introduced a reason without updating the classifier, OR the row's reason column was set by a code path that bypassed the helper. Open an issue with the reason string + audit log.
